# Plan Service

아이디어를 구현 가능한 제품 전략으로 정리하되 코드를 작성하지 않는다.

## 절차

1. `apps/registry.json`, 기존 앱의 `APP.md`, `docs/ARCHITECTURE.md`를 읽어 중복 서비스를 확인한다.
2. 사용자 문제, 검색 의도, 반복 사용 가능성, 운영 비용, 수익화 가능성을 기준으로 후보를 비교한다.
3. 고위험 가정이나 외부 계정·API·콘텐츠가 필요한 지점을 먼저 밝힌다.
4. 사용자 명령에서 실행 모드를 판별한다.
   - 기본값은 `staged`: Stitch 디자인을 생성한 뒤 사용자 확인을 기다린다.
   - "한번에", "중간 확인 없이", "디자인부터 구현까지", "끝까지 진행"을 명시하면 `one-shot`: 디자인 자체 검수 후 로컬 구현·검증까지 이어간다.
5. 사용자와 하나의 아이디어를 확정한다. 일괄 모드에서는 사용자가 판단을 위임한 세부 선택을 합리적으로 결정하되 제품 정체성이나 비용을 크게 바꾸는 선택은 묻는다.
6. [전략 출력 형식](references/strategy-output.md)에 따라 실행 모드를 포함해 한국어로 결과를 작성한다.
7. `$design-service`에 전략과 실행 모드를 handoff한다. `one-shot`이면 별도 디자인 승인 없이 `$build-service`까지 이어가도록 명시한다.

## 경계

- 이 skill 자체는 `pnpm create:app`, 파일 생성, dependency 설치, DB 변경을 실행하지 않는다. 일괄 모드의 실제 구현은 `$design-service`와 `$build-service` 절차가 이어서 담당한다.
- 검색량이나 경쟁 강도를 확인하지 않았다면 수치로 단정하지 않는다.
- 앱 하나에는 주된 검색 의도 하나를 부여한다.
- 기존 형제 앱과 기능이 겹치면 통합·차별화·폐기 중 적절한 방향을 제안한다.
